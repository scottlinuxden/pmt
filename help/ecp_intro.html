Engineering Change Proposal Instructions

Engineering Change Proposals (ECPs) are requested changes by any project team member to an established baseline such as functional, allocated, development, etc.  ECPs differ from System Problem Reports (SPRs), which record CI defects that require correction, ECPs are requests for a change to a CI that has been established from an approved baseline. 

-- Creation --


Note:  The originator will fill out the Creation portion.  The only exception will be the Status field that is first filled in by the Originator.  Once this is done, the appropriate personnel will change the Status field as the ECP goes through its processes.

Note:  Field Labels in Red or Bold on Monochrome Displays are required.

ECP Id:  This field will automatically be populated with the ECP Id.

Prefix:  Indicates which phase of development the change was proposed.  Choose one of the following from the drop down box.

·        OFP 1
·        OFP 2
·        OFP 3
·        OFP 4
·        OFP 5
·        OFP 6
·        OFP 7
·        OFP 8

Outside Id:  This is a text field where the issue initiator can insert their own issue tracking number.

Type:  This field indicates the type of change proposed.  Choose one of the following from the drop down box.

·        Design
·        Documentation
·        Hardware
·        Logic
·        Software

Change Name:  This should state a short description of the proposed change.

Status:  Lists the possible status conditions for the proposed change.  The status will evolve as the ECP is reviewed.  The personnel, as indicated within parenthesis, will change the Status field as indicated.  Choose one of the following from the drop down box.

·        Analysis Completed                        (Originator)
·        Analysis Required                            (Analyst)
·        Change Completed                          (Analyst)
·        Change Implemented                      (Test Manager)
·        Duplicate                                        (Analyst)                                   
·        Hold (will be considered later)         (Analyst)
·        Retire (not to be implemented)       (Analyst)
·        Under CCB investigation                 (CM Manager)          
·        Under GCCB investigation               (NASA DFRC CM Manager)
·        Under scope investigation                (CM Manager)

Change Description:  This field should contain the detailed description text of the proposed change.

Originator:  Indicates who initiated the ECP.  Each project member is selectable via a drop down box.  The initiator is required to select his or her name from the list when creating the ECP.

Origination Date:  The date the ECP was submitted.  This field is auto-populated, but can be changed if necessary.  The format required is MM-DD-YYYY.

--  Analysis Section  --


Analyst Username/Password:  The Analyst's signature is an indication that analysis of the change has been completed and the appropriate analysis results have been provided.  This also signals the Software Manager that a review of the Analysis is required.

Analyst Signature:  This field will be populated once the Analyst has entered a valid Username and Password, the option Sign has been selected from the Signature Function, and the ECP has been saved.

Signature Function:  This field allows the Analyst to Erase or Sign his name to the Analyst Signature above. 

·        Erase
·        Sign

Analysis:  This text field is where the Analyst who is working the change should document the investigation results.

Alternatives/Impacts:  This text field is where the Analyst should state any alternatives or system impacts which could result from the suggestions in the Change Description.  The Analyst would also list what the ECP would impact, i.e. baseline, documentation, etc. and whether there would be a greater or lesser impact to these items. 

Baseline Affected:  The Analyst should document which baseline would be affected by the change.

Documentation Affected:  The Analyst should state which documentation would be affected by the change.

Impacts on system:  The Analyst should state whether or not the change will require more system resources and how the change may impact the operation of the hardware.

Impacts on Systems Employment:  The Analyst should state what would be required to do the job, i.e. number of people, types of skills needed, and the time involved.

Impacts on System Resources:  The Analyst should list the necessary equipment required to complete the change.

Development Requirements:  The Analyst should list the requirements that will be added to or modified in the requirements specification.

Man-Hour Estimate:  The Analyst should state how many man-hours would be required to make the change.  The format is #####.

Date:  This is the date the Analyst entered the information required for analysis.  The format is MM-DD-YYYY.

Recommended Implementation Point:  The Analyst should list the recommended implementation point, i.e. time frame of year, in between specific builds, after deliver or before delivery.

Implementation Timing Dependencies:  The Analyst should state what other design and development dependencies will be impacted in making the change.

Related SPR or ECP Number:  The Analyst should state whether an SPR or another ECP is related to this ECP.  The format is #####.

Software Manager Username/Password:  An indication that the Software Manager has completed the review of the analysis.  The Software Manager's signature indicates the ECP is ready for CCB review.

Software Manager Signature:  This field will be populated once the Software Manager has entered a valid Username and Password, the option Sign has been selected from the Signature Function, and the ECP has been saved.

Signature Function:  This field allows the Software Manager to Erase or Sign his name to the Software Manager Signature above. 

·        Erase
·        Sign

-- Completion Approval  --

Software Manager Username/Password:  The Software Manager will only enter this information once the ECP has been approved by both the CCB and the GCCB and the ECP has been implemented.  It should not be signed if it has a disposition of Changes to ECP, Deferred, or Disapproved.    

Software Manager Signature:  This field will be populated once the Software Manager has entered a valid Username and Password, the option Sign has been selected from the Signature Function, and the ECP has been saved.

Signature Function:  This field allows the Software Manager to Erase or Sign his name to the Software Manager Signature above. 

·        Erase
·        Sign

QA Manager Username/Password:  The QA Manager will fill this in only when the ECP has been approved and completed.  They will check to see that all the fields have been filled in with some type of notation.  If something is N/A or is not affected, it should state this in each field.

QA Manager Signature:  This field will be populated once the QA Manager has entered a valid Username and Password, the option Sign has been selected from the Signature Function, and the ECP has been saved.

Signature Function:  This field allows the QA Manager to Erase or Sign his name to the QA Manager Signature above. 

·        Erase
·        Sign

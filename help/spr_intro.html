SYSTEM PROBLEM REPORTS

 

System Problem Reports (SPRs) will document any problems, errors, or discrepancies that are detected in the development of software, hardware, or documentation.  The generation of a SPR is not limited solely to software errors and may also be used to document problems or discrepancies in requirements, design, or other development documentation.

 

When a problem is encountered either in documentation review or during a test activity, the person identifying the problem will be responsible for logging onto the ISR Program website and using the web-based SPR interface to submit the SPR.  Field labels in red or bold on monochrome displays are required.  Once the SPR has been submitted to the system, the ISR Software Manager will disposition the SPR and, if necessary, assign a responsible Analyst to work it to resolution.

 

The following list provides the title of the position assigned to fill out specified fields.  Some text boxes will require more than one person to enter data as the SPR progresses.  The initials will precede the field name to show assigned personnel.

 

·       II - Issue Initiator

·       SM - Software Manager        

·       DV - Developer      

·       TM - Test Manager        

  • CM - CM Manager
  • QA - QA Manager

 

 

II - PREFIX: Via a drop down box, a field indicating in which phase of development the problem was identified. The options under the drop down box are:

 

  • Build 1
  • Build 2
  • Build 3
  • Critical Design Review
  • Formal Qualification Testing
  • Internal Testing
  • Internal Test Readiness Review
  • Preliminary Design Review
  • Prototype Feedback
  • System Requirements Review
  • Technical Interchange Meeting
  • Test Readiness Review

 

 

II - OUTSIDE ID: A text field where the issue initiator can insert their own issue tracking number (i.e. John Doe issue #1 could be represented as Doe-001).

 

II - PROBLEM GIST: A text field providing a short summary of the problem statement (i.e. the issue subject or title).

 

II - SOURCE: A field showing, via a drop down box, what organization that the issue initiator belongs to.

 

  • ISR
  • DFRC
  • Boeing

 

II, SM, DV - STATUS: A field listing the possible status conditions for the issue.  The SPR will undergo several status changes and can be changed via the drop down box.  The issue initiator, the Software Manager, and the Developer will all fill in this box.  Each SPR starts in a “New Problem” state and progresses through the various analysis and implementation stages until completed.  The complete list of options is as follows:

 

  • New Problem
  • Analysis Completed
  • Analysis Required
  • Corrective Action Completed
  • Duplicate
  • Fixed (complete)
  • Hold (will be considered later)
  • Retire (not to be implemented)
  • Unable to duplicate problem
  • Under CCB investigation
  • Under scope investigation

 

II - PROBLEM DESCRIPTION: A text field that contains the detailed description of the problem.

 

II - PROBLEM DUPLICATION:  An indication from the issue originator of whether the problem can be duplicated or not.  The options for this field are either Yes/No or Not Applicable.

 

II - SYSTEM STATUS: A drop down box of possible system states that existed at the time of the problem. 

 

·        Running

·        Halted

·        Unknown

 

II - PRIORITY: A determination of problem severity indicating the priority that should be attached in finding a solution.  The possible priorities are as follows:

 

  • Essential Function Degraded - No work around exists
  • Essential Function Degraded - Work around exists
  • Inconvenience
  • Other
  • Prevention of an essential function

 

II - CATEGORY: A field showing, via a drop down box, the class of problems to which the SPR belongs. 

 

  • Design Defect
  • Documentation Defect
  • Hardware Defect
  • Logic Defect
  • Software Defect

 

II - ORIGINATOR:  A field showing the SPR initiator.  Each project member is selectable via the drop down box that is provided and the initiator is required to select his or her name from the list when creating the SPR.

 

ORIGINATION DATE: The date that the SPR was submitted. This field is auto-populated.

 

SM - ASSIGNED TO: A field that the Software Manager populates with the responsible Analyst, which is selectable via a drop down box.  The responsible Analyst is the developer responsible for investigating the problem, documenting the problem analysis, and implementing a corrective action if needed.

 

==ANALYSIS APPROVAL==

 

DV - ANALYSIS:  A text field where the Analyst working the problem documents the results from the investigation. 

 

DV - ANALYST USERNAME/PASSWORD: An indication that the Analyst has completed the analysis of the problem and has provided the appropriate analysis results.  The Analyst’s signature here signals the Software Manager that his review of the Analysis is required prior to any implementation of a Corrective Action can be initiated.

 

SM - SOFTWARE MANAGER USERNAME/PASSWORD: An indication that the Software Manager has reviewed the Analysis and agrees that steps can proceed toward the implementation of a Corrective Action.

 

DV - CORRECTIVE ACTION:  A text field that the Analyst populates with the necessary information to document the Corrective Action for this SPR.  After correcting the problem, the developer should insert a log message into each source code file within CVS that has been changed stating the SPR number and a short summation of the corrective action taken.

 

DV - CONFIGURATION ITEMS IMPACTED:  A text field to indicate the configuration items impacted by this problem and its associated corrective actions.

 

TM - TEST PLAN: A text field to indicate what changes, if any, are needed to the test plan.

 

TM - TEST RESULTS: A text field that documents the test results showing that the problem has been resolved.

 

==COMPLETION APPROVAL==

 

SM - SOFTWARE MANAGER USERNAME/PASSWORD:  An indication that the Software Manager has reviewed the Analysis and Corrective Action and agrees with the closure of the SPR.

 

TM - TEST MANAGER USERNAME/PASSWORD: An indication that the Test Manager has reviewed the Analysis and Corrective Action and agrees with the closure of the SPR.

 

CM - CM MANAGER USERNAME/PASSWORD: An indication that the CM Manager has reviewed the Analysis and Corrective Action and agrees with the closure of the SPR.

 

QA - QA MANAGER USERNAME/PASSWORD: An indication that the QA Manager has reviewed the Analysis and Corrective Action and agrees with the closure of the SPR.

 

 

Sample SPR

 

Upon finding a problem, software, hardware, or documentation, the issue initiator will log on to the IFC Project Website and select Problem Report from the menu.  He or she will then fill in their Username and Password and select Query.  From this point, the initiator will be able to view all existing SPRs or Create a new one. 

 

To create a new SPR, the initiator will select the Create button and a blank form will appear.  The issue initiator will fill in the following fields:

 

·        Prefix

·        Outside ID (if desired)

·        Problem Gist

·        Source

·        Status  (The originator should only select the New Problem option.)

·        Problem Description

·        Problem Duplication  (This would not apply to documentation.)

·        System Status

·        Priority

·        Category

·        Originator

 

Then the Software Manager will fill in the following fields:

 

·        Status 

o       Duplicate, Hold, Retire  (No action required for this SPR.)

o       Under CCB Investigation  (This option may be selected prior to the determination that analysis is required or may be selected upon completion of analysis.)

o       Under Scope Investigation  (This option is selected when the problem may or may not be applicable to the software.)

o       Analysis Required  (SM will assign Developer to work the issue.)

·        Assigned To  (Only when the appropriate developer has been selected.)

 

Once the developer has been assigned the problem, he or she will perform the analysis and enter it into the SPR form for Software Manager approval.  Upon Software Manager approval, the Developer would then detail the Corrective Action and the impacted configuration items and the Test Manager will indicate what changes, if any, are required in the test plan.  The Test Manager and the Developer will implement the corrective action and produce test results verifying issue closer.  If the analysis is not approved, it would go back to the Developer for further analysis.

 

Following completion of problem resolution, the Software Manager, Test Manager, Configuration Manager, and Quality Assurance Manager will review the issue status and closure and if satisfied, complete the signature section.
